## Overview

PixelFlow is designed for high-performance client-side image processing. This guide covers batch processing strategies, configuration tuning, memory optimization, and best practices to ensure smooth operation even with large image sets.

## Batch Processing

### Upload Batching

Images are uploaded in batches to prevent overwhelming the browser's memory and event loop:

```js src/lib/constants.js:12
// Number of images to load simultaneously per upload batch
export const UPLOAD_BATCH_SIZE = 3;
```

#### How Upload Batching Works

```js src/hooks/useImageUpload.js:33-107
const handleUpload = async (files) => {
  const TOTAL_BATCHES = Math.ceil(files.length / UPLOAD_BATCH_SIZE);
  const newImages = [];

  // Process images in batches
  for (let batchIndex = 0; batchIndex < TOTAL_BATCHES; batchIndex++) {
    const startIndex = batchIndex * UPLOAD_BATCH_SIZE;
    const endIndex = Math.min(startIndex + UPLOAD_BATCH_SIZE, files.length);
    const currentBatch = files.slice(startIndex, endIndex);

    // Show progress for large uploads
    if (files.length > UPLOAD_PROGRESS_THRESHOLD) {
      message.loading({
        content: `Processing images ${startIndex + 1}-${endIndex} of ${files.length}...`,
        key: 'uploadProgress',
        duration: 0,
      });
    }

    // Process current batch in parallel
    const batchResults = await Promise.all(
      currentBatch.map(async (file, idx) => {
        // Create preview, estimate memory, etc.
        // ...
      })
    );

    newImages.push(...batchResults.filter(Boolean));

    // Give browser time to update UI
    await new Promise((resolve) => setTimeout(resolve, 10));
  }

  setUploadedImages(newImages);
};
```

<Note>
  Files **within each batch** are processed in **parallel** using `Promise.all()`, but **batches themselves** are processed **sequentially**.
</Note>

### Processing Batching

```js src/lib/constants.js:15
// Number of images to process simultaneously per processing batch
export const PROCESSING_BATCH_SIZE = 3;
```

This controls how many images are processed through the optimization/resize/crop/format pipeline at once.

### Download Batching

```js src/lib/constants.js:18
// Number of images to prepare simultaneously per download batch
export const DOWNLOAD_BATCH_SIZE = 5;
```

When creating ZIP files or bulk downloads, images are prepared in batches of 5 to avoid memory spikes.

### Progress Indication

```js src/lib/constants.js:21
// Minimum files to show progress indicator during upload
export const UPLOAD_PROGRESS_THRESHOLD = 5;
```

Progress messages only appear when uploading **5 or more** images to reduce UI noise for small batches.

<Tip>
  **Tuning tip**: Increase batch sizes (e.g., to 5-7) on high-memory devices, or decrease (to 1-2) on mobile devices with limited RAM.
</Tip>

## Image Optimization Settings

### Default Optimization Parameters

PixelFlow uses `browser-image-compression` with conservative defaults:

```js src/lib/constants.js:44-47
// Maximum size of optimized file (MB)
export const OPTIMIZE_MAX_SIZE_MB = 1;

// Maximum dimension (width or height) of optimized image (pixels)
export const OPTIMIZE_MAX_DIMENSION = 1920;
```

#### Optimization Usage

```js Example from processing pipeline
import imageCompression from 'browser-image-compression';
import { OPTIMIZE_MAX_SIZE_MB, OPTIMIZE_MAX_DIMENSION } from '@/lib/constants';

const options = {
  maxSizeMB: OPTIMIZE_MAX_SIZE_MB,        // 1 MB
  maxWidthOrHeight: OPTIMIZE_MAX_DIMENSION, // 1920 px
  useWebWorker: true,  // Offload to worker thread
};

const compressedFile = await imageCompression(file, options);
```

<Warning>
  Setting `OPTIMIZE_MAX_SIZE_MB` too low (e.g., 0.5 MB) may result in visible quality loss for high-resolution images.
</Warning>

## Quality Settings

Different operations use different quality levels:

```js src/lib/constants.js:69-79

// Resize quality (react-image-file-resizer) - 0 to 100

// Memory compression quality - 0 to 1
export const MEMORY_COMPRESSION_QUALITY = 0.85;

// Low-res preview quality - 0 to 1
export const LOW_RES_PREVIEW_QUALITY = 0.5;
```

### Quality Comparison

| Operation | Quality | Range | Purpose |
|-----------|---------|-------|----------|
| Format Conversion | `0.92` | 0-1 | High quality for user downloads |
| Resize | `100` | 0-100 | Maximum quality during resize |
| Memory Compression | `0.85` | 0-1 | Balance quality/memory |
| Low-Res Preview | `0.5` | 0-1 | Fast preview for large files |

<Tip>
  **Performance tip**: Reducing `FORMAT_CONVERSION_QUALITY` to `0.85` can significantly speed up conversions with minimal visual impact.
</Tip>

## Memory Optimization

### Large File Handling

PixelFlow automatically optimizes memory usage for large files:

```js src/lib/constants.js:28
// Files larger than 10 MB are considered "large"
export const LARGE_IMAGE_THRESHOLD = 10 * 1024 * 1024;
```

When a file exceeds this threshold:
1. **Low-res preview** is created (300px × quality 0.5) instead of full preview
2. **Memory estimation** is performed
3. **Warning logged** if memory usage exceeds 500 MB

### Memory Thresholds

```js src/lib/constants.js:31
// Trigger memory warning at 500 MB estimated usage
export const MEMORY_WARNING_THRESHOLD = 500 * 1024 * 1024;
```

### Memory-Constrained Dimensions

```js src/lib/constants.js:86-89
// Max dimensions when compressing for memory conservation
export const MEMORY_MAX_WIDTH = 1920;
export const MEMORY_MAX_HEIGHT = 1080;
```

Images exceeding these dimensions are automatically downscaled during memory-constrained operations.

## Processing Pipeline Order

PixelFlow processes images in a specific order for optimal performance:

```mermaid
graph LR
    A[Original Image] --> B[1. Optimize]
    B --> C[2. Resize]
    C --> D[3. Crop]
    D --> E[4. Format Convert]
    E --> F[Final Image]
```

### Why This Order Matters

<Steps>
  <Step title="Optimize First">
    Reduces file size and dimensions early, making subsequent operations faster
  </Step>
  
  <Step title="Resize Second">
    Resizing after optimization works with smaller files
  </Step>
  
  <Step title="Crop Third">
    Cropping after resize ensures correct final dimensions
  </Step>
  
  <Step title="Format Convert Last">
    Final conversion happens once, on the fully processed image
  </Step>
</Steps>

## Web Workers

PixelFlow leverages Web Workers where possible:

```js
import imageCompression from 'browser-image-compression';

const options = {
  maxSizeMB: OPTIMIZE_MAX_SIZE_MB,
  maxWidthOrHeight: OPTIMIZE_MAX_DIMENSION,
  useWebWorker: true,  // Offload to background thread
};

const compressed = await imageCompression(file, options);
```

<Note>
  The `browser-image-compression` library automatically spawns Web Workers when `useWebWorker: true`, keeping the main thread responsive during compression.
</Note>

## Canvas Optimization

### Canvas Size Limits

Most browsers limit canvas size to prevent memory exhaustion:

| Browser | Max Dimension | Max Area |
|---------|---------------|----------|
| Chrome | ~32,767 px | ~268 MP |
| Firefox | ~32,767 px | ~124 MP |
| Safari | ~4,096 px | ~16 MP |

<Warning>
  Safari has the **most restrictive** canvas limits. Images exceeding 4,096px on any side may fail to render on iOS/macOS.
</Warning>

### Chunking for Large Images

For images exceeding canvas limits, use chunking:

```js src/lib/constants.js:95
// Chunk size when splitting extremely large images
export const IMAGE_CHUNK_SIZE = 1024;
```

```js
import { splitImageIntoChunks } from '@/lib/memoryManager';

const chunks = splitImageIntoChunks(largeImage, IMAGE_CHUNK_SIZE);

// Process each chunk separately
chunks.forEach(chunk => {
  // Process chunk.canvas
});
```

## Performance Monitoring

### Memory Estimation

```js

import { MEMORY_WARNING_THRESHOLD } from '@/lib/constants';

const estimatedMemory = estimateImageMemoryUsage(file);
```

## Common Performance Issues

### Issue: Browser Freezes During Upload

**Cause**: Batch size too large or files too big  
**Solution**: Reduce `UPLOAD_BATCH_SIZE` to 1-2 and enable progress indicators

### Issue: Out of Memory Errors

**Cause**: Processing too many large images simultaneously  
**Solution**: Reduce `PROCESSING_BATCH_SIZE` and lower `MEMORY_MAX_WIDTH`/`MEMORY_MAX_HEIGHT`

### Issue: Slow Format Conversions

**Cause**: High quality settings on large images  
**Solution**: Reduce `FORMAT_CONVERSION_QUALITY` to 0.85 and optimize images first

### Issue: Preview Generation Lag

**Cause**: Creating full-res previews for large files  
**Solution**: Lower `LARGE_IMAGE_THRESHOLD` to 5MB to trigger low-res previews earlier

## Benchmarking Results

Typical performance on a mid-range desktop (2020 MacBook Pro):

| Operation | 1 Image (5MB) | 10 Images (50MB) | 100 Images (500MB) |
|-----------|---------------|------------------|--------------------|
| Upload | ~200ms | ~2s | ~25s |
| Optimize | ~800ms | ~8s | ~90s |
| Resize | ~150ms | ~1.5s | ~18s |
| Format Convert | ~100ms | ~1s | ~12s |
| ZIP Download | ~500ms | ~4s | ~45s |

<Tip>
  Enable `useWebWorker: true` to reduce these times by **30-50%** on multi-core systems.
</Tip>
